3.2 - In a recent project, I was tasked with setting up OpenSearch's new semantic search functionality at scale, which required bringing our own embeddings rather than using a fully managed, out-of-the-box solution. The challenge was to strike the right balance between retrieval accuracy, latency, and cost given real-world production needs.

We evaluated several embedding models ranging from different provider to different parameter sizes, testing each for semantic accuracy with our datasets—higher-quality models captured more nuance and improved end-user search relevance, but they also tended to introduce greater latency and increased resource requirements. For example, large language model-based embeddings significantly outperformed lightweight alternatives in accuracy, but their query time and hardware costs made them prohibitive for low-latency, high-traffic use cases. Cost modeling was straightforward—just infrastructure and model hosting. However, as we explored third-party embedding APIs and some LLM options, token-based pricing introduced complexity: costs became highly variable, dependent on input length and usage spikes. I architected the pipeline to include maximum input truncation, batch processing, and a caching layer for common queries—improving latency and predictability as well. The result was that we were able to choose what was acceptable and at a cost range of which we can accept.

c. Application and model security: My main focus is on setting up the inbound and outbound data flows to ensure that all data entering and leaving the system is strictly validated and properly encoded. I configure strong guardrails on agents, setting clear boundaries for what they can access or perform. At the infrastructure level, I make sure firewalls are set up correctly to control and monitor network traffic, and that everything transmitted over the network is encoded or encrypted to prevent leaks or tampering. Additionally, I enforce policies so that only safe, validated data can be acted upon by agents or models, minimizing risk from both input and output. These controls are documented and consistently included in onboarding, code reviews, and operational checklists to make robust security standard practice across the team.

## Architecture & Leadership Scenario (choose one)

5. To turn a multi-step agent that runs for minutes into a production-ready API product, I would design the system around REST endpoints, leveraging RabbitMQ as the message queue for orchestrating agent execution and Redis for caching responses and run status updates. When a client initiates a run via a `POST /runs` request, the request parameters are validated, a new job is enqueued into RabbitMQ, and a unique `run_id` is generated and immediately returned to the client. Workers consume jobs from RabbitMQ and, as the agent progresses through its steps, they continually update the current status, partial results, or final outputs in Redis. This approach provides efficient, low-latency status checks via `GET /runs/{run_id}`, as Redis can quickly return up-to-date information about a run’s progress or completion without hitting the primary database or blocking on computation. All state transitions (e.g., `PENDING`, `RUNNING`, `PARTIAL_RESULT`, `SUCCEEDED`, `FAILED`) and cost or budget checks are managed within the worker process and reflected in Redis. This pattern handles timeouts and scalability concerns elegantly, since no REST call blocks beyond initial validation—clients poll for status or results as needed, and Redis provides fast intermediate and final data access. Comprehensive logging and observability are also layered in by emitting structured events as each agent progresses, and legacy synchronous flows can be migrated over time by wrapping their logic to interact with the new REST+queue-backed system. By combining REST endpoints with RabbitMQ and Redis, this design delivers an efficient, resilient, and scalable API product for long-running agents, with robust caching, status tracking, and a smooth migration path for users.
